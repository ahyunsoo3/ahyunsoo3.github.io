AI Pipeline

Four AI providers handle speech, text, evaluation, and conversation. All AI calls go through an edge function credit gate — the client never holds an API key. Requests are retried up to 10× with exponential backoff.

AI call flow

The Flutter client sends a JWT-authenticated POST to the relevant edge function. The edge function checks the user's credit balance, calls the upstream AI provider, deducts credits on success, and returns the result. Provider failures trigger retry. 

Flutter Client
└── POST /functions/v1/{ai-fn}  (JWT + payload)
    │
    ├── _shared/auth.ts → verify JWT
    ├── DB: SELECT user_credits WHERE user_id = $uid
    │       └─ insufficient? → 402 Payment Required
    │
    ├── Call upstream provider  (retry ×10, expo backoff)
    │   ├── tts-gemini       → Gemini 2.5 Pro TTS
    │   ├── stt-deepgram     → Deepgram Nova-3 STT
    │   ├── evaluate-writing → ChatGPT-4o (structured output)
    │   └── generate-study-plan → Gemini 2.5 Pro (JSON mode)
    │
    ├── DB: UPDATE user_credits  (deduct on success)
    ├── DB: INSERT practice_history
    └── Response → Flutter Client

Step-by-step walkthrough

Every AI request follows this exact pipeline. The credit gate and retry logic are encapsulated inside the edge function — the Flutter client sees only success or a typed error.

1
Client sends authenticated POST
Flutter's core_network Dio client sends POST /functions/v1/{fn-name} with the user's JWT in the Authorization header and the payload (text / audio / prompt) in the body. Timeout: 30s.
2
JWT verified
_shared/auth.ts · verifyJwt(req) calls Supabase Auth to validate the token and extract user_id. Invalid or expired tokens return 401 Unauthorized immediately — no DB queries executed.
3
Credit balance checked
SELECT balance FROM user_credits WHERE id = $uid. If balance < required_cost, the function returns 402 Payment Required with the current balance and the cost in the response body. No AI call is made.
4
Upstream provider called (with retry)
The edge function calls the AI provider with exponential backoff. Attempt 1 at 0s, attempt 2 at 1s, attempt 3 at 2s, …, up to the plan's max retry count (3–10). Provider-specific errors (rate limit, timeout) trigger retry; malformed input errors do not.
5
Credits deducted (on success only)
UPDATE user_credits SET balance = balance - $cost WHERE id = $uid. This only executes after a successful provider response. If the provider fails after all retries, the user is never charged. Deduction and history insert happen in a single database transaction.
6
Session logged
INSERT INTO practice_history (user_id, module, type, score, ai_feedback). The AI response is stored as structured JSON in ai_feedback, enabling the analytics module to display historical scores without re-calling the AI.
7
Response returned to client
The processed result (transcript, rubric JSON, audio bytes, or study plan) is returned as 200 OK. The Riverpod notifier updates state and Flutter rebuilds the feedback screen.

Prompt design

Each AI use case has a carefully structured system prompt that enforces output format, scoring rubric, and tone. All prompts are stored in the edge function — never in the client.

evaluate-writing — system prompt excerpt
You are an expert TOEFL iBT writing evaluator. Score the essay on the four iBT criteria,
each on a 0–30 scale. Return ONLY valid JSON, no prose.

Response schema:
{
  "taskAchievement": { "score": number, "feedback": string },
  "coherenceCohesion": { "score": number, "feedback": string },
  "lexicalResource": { "score": number, "feedback": string },
  "grammaticalRange": { "score": number, "feedback": string },
  "overallScore": number,
  "strengths": string[],
  "improvements": string[]
}

Task prompt: {{task_prompt}}
Essay: {{essay_text}}
generate-study-plan — system prompt excerpt
You are a TOEFL study planner. Given the student's data, create a personalized weekly
study plan as structured JSON. Distribute tasks to reach the target score by the exam date.

Student data:
- Current scores: R={{reading}} L={{listening}} S={{speaking}} W={{writing}}
- Target score: {{target_score}} by {{exam_date}}
- Weak sections: {{weak_sections}}
- Daily availability: {{daily_hours}} hours

Return a JSON array of 7 days, each with an array of tasks:
{ "day": "Monday", "tasks": [{ "module": string, "duration_min": number, "focus": string }] }
T
Structured output enforced at prompt level All Gemini and GPT-4o prompts use explicit JSON schema instructions in the system prompt, plus the provider's native structured-output mode (response_format: { type: "json_object" } for OpenAI, responseSchema for Gemini). This eliminates JSON parse failures in the edge function response handler.

Fallback behavior

When AI providers are unavailable or return errors, the system fails gracefully with informative user feedback and zero credit deduction.

ScenarioHTTPCredits chargedUser experience
Insufficient credits402None Upgrade prompt shown immediately; current balance displayed
Provider rate limit (429)—None Retried automatically with backoff; user sees a loading state
Provider timeout (>30s)—None Retry attempt; after max retries, 503 returned with "Try again later"
All retries exhausted503None "Service temporarily unavailable" screen with retry button
Invalid AI output (parse fail)422None Retry with slightly adjusted prompt; if persistent, 503
Provider returns success200Deducted Full feedback screen rendered
D
Why four different AI providers? Each provider is chosen for its strength in a specific task. Deepgram Nova-3 leads for real-time speech recognition accuracy. Gemini 2.5 Pro excels at structured JSON generation and multilingual TTS. GPT-4o's structured output mode produces the most consistent writing rubric scores. Inworld AI provides interactive NPC conversation that no general LLM matches for character immersion.
