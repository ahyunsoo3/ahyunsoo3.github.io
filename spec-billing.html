Purchase flow steps

Every purchase follows a server-verification round-trip before any credits are granted. The Flutter client handles the store UI, but all financial logic lives server-side.

1
Fetch products
BillingService.getProducts() queries the active store (App Store or Google Play) for available SKUs. Product details (name, price, description) are displayed in the billing screen.
2
User initiates purchase
User taps "Subscribe" or "Buy pack". core_billing calls the native store API. The platform presents its native payment sheet — the app has no access to payment details.
3
Platform confirms & returns receipt
The store processes payment and returns a purchase receipt (App Store) or purchase token (Google Play) to the app. At this point, the purchase is pending — no credits have been granted yet.
4
Edge function verifies receipt
App sends receipt + user JWT to POST /functions/v1/verify-{platform}. The edge function calls the platform's server API to confirm validity. This prevents receipt spoofing — the receipt is never trusted at face value.
5
Nonce checked for idempotency
Before inserting, the edge function checks billing_nonces for a matching nonce. If found (already processed), it returns the existing result without double-granting. This handles network retries and replay attacks.
6
Subscription inserted & credits granted
INSERT INTO subscriptions and UPDATE user_credits happen atomically. The nonce is marked used = true. Monthly credit allowance is set according to the plan tier.
7
UI updated
Edge function returns 200 OK. The billing module's Riverpod notifier invalidates the plan cache, the app navigates to the confirmation screen, and the new credit balance is reflected immediately.

Anti-fraud & idempotency

The billing system must handle network failures, retries, and malicious replay attempts without ever double-charging or double-crediting a user.

N
billing_nonces table Every purchase verification request includes a client-generated nonce (UUID). Before processing, the edge function checks SELECT 1 FROM billing_nonces WHERE nonce = $nonce AND used = true. If found, the function returns the cached successful result without re-processing. The nonces table has a UNIQUE constraint on the nonce column and is only accessible via the service role — clients cannot read or modify it.
Replay attack prevention
  • Receipt nonces are single-use: once a nonce is marked used = true, any subsequent request with the same nonce returns the original result
  • Nonces expire after 72 hours — old receipts cannot be replayed indefinitely
  • Lemon Squeezy webhooks use HMAC-SHA256 signature verification instead of nonces
Double-credit prevention
  • Credit grant and subscription insert are wrapped in a PostgreSQL transaction — partial writes are impossible
  • subscriptions.expiry is checked before granting renewal credits — no credits if existing subscription is still active
  • grant-credits function is called only after the billing function confirms the transaction is new
Cross-device restore
  • restore-purchase uses UPSERT on subscription primary key — safe to call multiple times
  • Credits are only topped up if balance is below the plan's monthly allowance
  • Restore flow is triggered automatically on first launch after sign-in on a new device
Platform-specific notes
  • App Store: Uses App Store Server Notifications (v2) for real-time subscription status updates
  • Google Play: Developer Notifications via Pub/Sub for renewal and cancellation events
  • Lemon Squeezy: Webhook-based; no mobile SDK required (desktop / web only)
