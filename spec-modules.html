Flutter Modules

The client codebase is split into 14 independent feature modules and 3 shared packages, following a clean feature-first architecture. Each module owns its routes, state, and repository.

14 feature modules

Each feature module has its own lib/ tree and registers routes into the root GoRouter shell. Cross-module calls go through Riverpod providers — never direct class imports. 

lib/
├── core/
│   ├── core_ui/          (design system)
│   ├── core_network/     (Dio + interceptors)
│   └── core_billing/     (IAP abstraction)
│
└── features/
    ├── auth/             01 · Sign-in / magic-link / OAuth
    ├── onboarding/       02 · Goal setup, diagnostic test
    ├── home/             03 · Dashboard, streak, quick actions
    ├── speaking/         04 · AI Speaking practice (Deepgram)
    ├── writing/          05 · AI Writing evaluation (ChatGPT-4o)
    ├── listening/        06 · Audio comprehension exercises

Architecture rules

!
No direct cross-module imports Feature modules must never import classes directly from other feature modules. All cross-module communication goes through Riverpod providers. This is enforced by analysis_options.yaml linting rules and checked in CI.
T
Repository pattern everywhere Every module has a *Repository class that owns all Supabase calls. Riverpod AsyncNotifiers call only the repository — never the Supabase client directly. This makes testing straightforward: mock the repository, test the notifier.
N
Shared packages, not shared modules Common UI components, network infrastructure, and billing logic live in core_ui, core_network, and core_billing packages — not inside any feature module. A feature module should be removable without breaking any other module.

Routing structure

GoRouter 14 with nested shell routes. The root ShellRoute owns the bottom navigation bar. Each feature module registers its routes into the shell using a GoRoute extension method.

GoRouter shell structure (Dart)
GoRouter(
  routes: [
    ShellRoute(                  // bottom nav shell
      builder: (ctx, state, child) => HomeShell(child: child),
      routes: [
        ...AuthRoutes.routes,        // /login, /magic-link
        ...HomeRoutes.routes,        // /home
        ...SpeakingRoutes.routes,    // /speaking, /speaking/:sessionId
        ...WritingRoutes.routes,     // /writing, /writing/:taskId
        ...ListeningRoutes.routes,   // /listening, /listening/:exerciseId
        ...ReadingRoutes.routes,     // /reading, /reading/:passageId
        ...VocabRoutes.routes,       // /vocabulary
        ...MockTestRoutes.routes,    // /mock-test, /mock-test/:testId
        ...AiTutorRoutes.routes,     // /ai-tutor, /ai-tutor/:sessionId
        ...AnalyticsRoutes.routes,   // /analytics
        ...StudyPlanRoutes.routes,   // /study-plan
        ...SettingsRoutes.routes,    // /settings, /settings/profile
        ...BillingRoutes.routes,     // /billing, /billing/plans
      ],
    ),
  ],
)
T
Deep link ready Every route has a named path, enabling deep links from push notifications (e.g. /speaking/session-123 opens a specific practice session directly). GoRouter handles the auth redirect automatically — unauthenticated deep links redirect to /login then resume the original destination after sign-in.
