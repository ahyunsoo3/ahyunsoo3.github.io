Review standards

What I check before merge

A checklist I run on every critical path — client, Edge Function, and ingestion script. If any item fails, the change does not ship until the failure mode is named and handled.

Security & auth

  • JWT verified on every Edge Function entry
  • RLS policies match the query shape — no broad SELECT * without filter
  • Service-role keys only in Deno env, never in Flutter assets
  • User ID from token, never from request body alone

Billing & webhooks

  • Nonce issued before purchase, deleted after successful verify
  • Store API called server-side — client token is evidence, not truth
  • Webhook handlers keyed by event ID for idempotency
  • Subscription period reset logic tested for edge dates

AI & cost

  • Credits deducted before the API call, not after
  • Input/output token caps enforced per tier
  • Retry loop has max attempts and visible user progress
  • Every provider call logs to record_ai_token_usage

Data & content

  • Pydantic validates JSONL before Postgres upsert
  • Stable slug keys — re-ingestion updates, not duplicates
  • Migrations backward-compatible with live app versions
  • Force-update gate when schema contract breaks

supabase/functions/verify_google_play_purchase/index.ts Blocked until fixed 
const { purchaseToken, nonceId } = await req.json();
+ const nonce = await getNonce(nonceId, userId);
+ if (!nonce || nonce.expires_at < new Date()) throw 400;
const receipt = await googlePlay.verify(purchaseToken);
+ await deleteNonce(nonceId);
await updateUserProfile(userId, receipt);
Review · billing

Without nonce deletion before profile update, a retried webhook or replayed client request can grant credits twice. The nonce must be consumed atomically with verification success.

Resolution · shipped

Nonce deleted immediately after validation, before store API call completes profile write. Purpose field scopes nonce to google_play_plan vs pack types.

Review · AI cost

Deducting credit after a successful response invites free retries on transient failures — users get 10 API calls for one credit. Pre-deduction is intentional: one credit, one scoring session.

Review · security

Accepting user_id from the request body lets any authenticated user trigger scoring on another account's quota. Identity must come from the verified JWT only.

All Edge Functions extract sub from Supabase JWT. RLS on user_profiles is the second layer; the function never trusts client-supplied IDs for writes.

Solo review loop

No second reviewer — stricter first pass

When you are the only engineer, review is how you simulate a senior teammate. I write the spec, implement, then re-read the diff as if I did not write it.

1
Diff against the failure mode

What breaks if this runs twice, if the API times out, or if the user is on Free tier?

2
Trace end-to-end

Flutter state → Edge Function → Postgres row → webhook echo. One broken hop means a support email.

3
Ship the smallest fix

Patch the root cause, verify on device + Supabase logs, then release. No drive-by refactors in hot paths.
