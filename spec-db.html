Database Schema

PostgreSQL database hosted on Supabase. Every table is protected by Row-Level Security (RLS) policies. No service-role key is ever exposed to the client.

SQL schema examples

Condensed DDL for the most important tables. JSONB columns are used wherever structured data needs flexibility without a full schema migration.

practice_history — core session table
CREATE TABLE practice_history (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  module      TEXT NOT NULL,          -- 'speaking' | 'writing' | 'listening' | 'reading'
  type        TEXT NOT NULL,          -- 'practice' | 'evaluation' | 'ai_feedback'
  score       NUMERIC(5,2),           -- 0–30 iBT scale (nullable for non-scored sessions)
  ai_feedback JSONB,                  -- full AI rubric or transcript JSON
  duration_s  INTEGER,                -- session length in seconds
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_practice_history_user_created
  ON practice_history (user_id, created_at DESC);
user_credits — credit balance tracking
CREATE TABLE user_credits (
  id             UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  balance        INTEGER NOT NULL DEFAULT 30 CHECK (balance >= 0),
  lifetime_used  INTEGER NOT NULL DEFAULT 0,
  updated_at     TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Trigger: update updated_at on every change
CREATE TRIGGER set_user_credits_updated_at
  BEFORE UPDATE ON user_credits
  FOR EACH ROW EXECUTE FUNCTION moddatetime();
billing_nonces — idempotency table
CREATE TABLE billing_nonces (
  id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id    UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  nonce      TEXT NOT NULL,
  used       BOOLEAN NOT NULL DEFAULT false,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX idx_billing_nonces_nonce ON billing_nonces (nonce);

-- Auto-cleanup: nonces older than 72h are deleted by a cron job
CREATE INDEX idx_billing_nonces_created ON billing_nonces (created_at);
vocabulary_progress — SM-2 spaced repetition
CREATE TABLE vocabulary_progress (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  word_id      UUID NOT NULL REFERENCES vocabulary_words(id),
  easiness     NUMERIC(4,2) NOT NULL DEFAULT 2.5,  -- SM-2 easiness factor
  interval     INTEGER NOT NULL DEFAULT 1,          -- days until next review
  repetitions  INTEGER NOT NULL DEFAULT 0,          -- consecutive correct answers
  next_review  DATE NOT NULL DEFAULT CURRENT_DATE,
  UNIQUE (user_id, word_id)
);

RLS policy examples

Every user-facing table uses a standard auth.uid() policy pattern. Content tables allow read access to all authenticated users but no writes from the client.

Standard user-scoped RLS (practice_history)
-- Enable RLS on the table
ALTER TABLE practice_history ENABLE ROW LEVEL SECURITY;

-- SELECT: users can only see their own rows
CREATE POLICY "Users can view own practice history"
  ON practice_history FOR SELECT
  USING (auth.uid() = user_id);

-- INSERT: users can only insert rows for themselves
CREATE POLICY "Users can insert own practice history"
  ON practice_history FOR INSERT
  WITH CHECK (auth.uid() = user_id);

-- No UPDATE or DELETE from client — sessions are immutable once created
Content table — read-only for authenticated users
ALTER TABLE questions ENABLE ROW LEVEL SECURITY;

-- All authenticated users can read questions (no user_id filter)
CREATE POLICY "Authenticated users can read questions"
  ON questions FOR SELECT
  USING (auth.role() = 'authenticated');

-- No INSERT, UPDATE, or DELETE — content is managed via admin dashboard only
billing_nonces — service role only
ALTER TABLE billing_nonces ENABLE ROW LEVEL SECURITY;

-- NO policies defined → default deny for all client roles
-- Only the service-role key (used inside edge functions) can read/write this table

-- This ensures no client can check or manipulate nonces directly

Table relationships

All user-facing tables have a foreign key to auth.users(id) with ON DELETE CASCADE — deleting a user removes all their data automatically.

auth.users 1 → N user_profiles ON DELETE CASCADE
auth.users 1 → N practice_history ON DELETE CASCADE
auth.users 1 → N mock_test_results ON DELETE CASCADE
auth.users 1 → 1 user_credits PK = user_id; ON DELETE CASCADE
auth.users 1 → N subscriptions ON DELETE CASCADE
auth.users 1 → N vocabulary_progress ON DELETE CASCADE · UNIQUE (user_id, word_id)
vocabulary_words 1 → N vocabulary_progress word_id FK

Data lifecycle

Data retained indefinitely
  • practice_history — all past sessions kept for analytics
  • mock_test_results — kept for score trend charts
  • subscriptions — kept for audit trail even after expiry
  • vocabulary_progress — SM-2 state must persist for the algorithm
Auto-cleaned data
  • billing_nonces — deleted after 72 hours via pg_cron
  • ai_tutor_sessions — optionally pruned after 90 days (configurable)
  • Supabase Storage audio files — signed URL cache expires in 24 hours
User-deleted data
  • Account deletion cascades to all user-owned tables
  • Supabase Storage bucket objects are deleted via a database trigger
  • Auth data (email, provider tokens) is deleted by Supabase Auth
Content data
  • questions, passages, vocabulary_words are managed via admin tooling only
  • New questions are added via Python scripts → CSV → Supabase bulk import
  • No client can modify content tables — enforced by RLS
