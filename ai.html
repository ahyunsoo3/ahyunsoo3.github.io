Gemini AI scoring

Writing and speaking, graded by Gemini

Not just "call the API and show the result." The scoring system has token budgets, per-tier model routing, retry loops, pre-call credit deduction, and async cost logging.

Writing

Email & academic discussion scoring

When a user submits a writing response, the flow is:

  1. Plan limit checked — does this tier allow AI scoring?
  2. 1 AI credit deducted from user_profiles.ai_scoring_credits before the API call — no free retries on failure.
  3. Input trimmed to ≤ 1,000 tokens (≈4 chars/token estimate) to cap cost.
  4. Model selected: gemini-3.1-flash-lite-preview for Free/Plus/Pro; gemini-3.1-pro-preview for Hyper tier only.
  5. Gemini generateContent called with output capped at 1,000 tokens.
  6. Response parsed into WritingEmailAssessment (band score, strengths, improvements, model answer).
  7. Result written to practice history. A retry progress callback drives the in-app loading UI.
Pre-call credit gate Token budget management Structured JSON output parsing
Speaking

Interview & listen-repeat scoring

Speaking scoring chains Deepgram STT → Gemini text assessment:

  1. User records audio → WAV bytes sent to DeepgramListenClient.transcribeAudioBytes().
  2. Deepgram Nova-3 returns transcript.
  3. Transcript fed to GeminiPracticeAssessmentClient for speaking-specific scoring rubric.
  4. Speaking assessment model: SpeakingInterviewAssessment — pronunciation, fluency, content, vocabulary bands.
  5. Token usage for both STT and LLM calls logged to record_ai_token_usage.

Gemini content generation

All 12 task types generated by AI

Practice content isn't written by hand. A Python pipeline prompts Gemini to output structured JSONL, validates each line with Pydantic, then loads it into Postgres.

1
Schema-first prompting

Each of 12 task types has a Pydantic model. Gemini is prompted with the schema as a system instruction and asked to output JSON Lines matching it exactly.

2
Structured output validation

Every line in the generated JSONL is parsed and validated by Pydantic. Schema violations, missing fields, and type mismatches are caught before any DB write.

3
Idempotent upsert

Content is keyed by a stable slug field. Re-running the ingestion script updates existing rows rather than duplicating them — safe to re-generate.

4
Model override

The generation model defaults to gemini-3.1-flash-lite-preview but can be overridden via GEMINI_MODEL env variable for quality A/B testing.

Cache-first lookup design

01

Context-aware selection

The lookup is not just the raw word. It includes the selected word plus nearby sentence context, so the model can return the right sense and register.

  • Handles ambiguous words by using surrounding text.
  • Keeps the definition appropriate for TOEFL study.
  • Returns a short gloss, register note, and minimal example.
02

Per-passage cache

Responses can be cached per passage, so repeat taps on the same reading content reuse the saved definition instead of calling ChatGPT again.

  • Reduces repeated latency while reading.
  • Reduces avoidable upstream API calls.
  • Keeps repeated lookup behavior predictable for users.
03

Credit-checked ChatGPT call

If the cache cannot answer, OpenAiTapDictionaryClient uses the same credit-checked AI path as the rest of the product.

  • Only misses need a new upstream AI request.
  • Input and output token usage are logged.
  • The new response becomes the cached answer for repeat taps.

Why the cache key includes context

// pseudo-Dart
String cacheKey(String word, String passageId, String sentence) {
  final context = nearbySentenceContext(sentence, around: word);
  return sha1([
    word.toLowerCase(),
    passageId,
    context,
  ].join('|'));
}

Including sentence context prevents incorrect reuse across senses. "bank" in finance and "bank" beside a river should not share the same cached gloss.

Credit & quota system

AI usage without cost surprises

Three independent credit types, each deducted before its AI call. Users can buy more via App Store, Google Play, or Lemon Squeezy. Quotas are enforced in the Flutter client before any network request is made.

AI scoring credits

1 credit consumed per Writing or Speaking AI scoring call. Deducted before the Gemini call — no refund on API failure. Packs: 40 or 100 credits.

ai_scoring_credits → Gemini scoring

AI tutor credits

1 credit per tutor chat message. Separate from scoring credits — users can buy tutor packs without scoring packs. Packs: 250 or 1,000 messages.

tutor_chat_credits → Gemini chat

Live caption seconds

Time-based credit for live speech captioning. Deducted per second of active captioning. Packs: 30 minutes or 2 hours.

live_captions_seconds → Deepgram streaming

How quota enforcement is designed

Client-side pre-check

PlanLimits checks the user's current credit balance before any UI that leads to an AI call. If the balance is zero, the action is gated with an upsell prompt — no wasted network round-trip.

Deduct before call

Credits are deducted from user_profiles before the Gemini or Deepgram call is made. This prevents users from hammering the API by retrying failed requests — the deduction is the gate.

Subscription period resets

For subscription tiers, the Lemon Squeezy webhook and App Store / Google Play verification functions check if a new billing period has started and reset the quota accordingly.

Usage logging

Actual input and output token counts from every Gemini and Deepgram call are logged to record_ai_token_usage with the model ID, timestamp, and user ID — enabling per-user cost analysis.
