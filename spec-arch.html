Full-stack topology

The Flutter client is the only user-facing artifact. Every external call routes through Supabase (auth, DB, storage, edge functions) or directly to a third-party API. AI responses are guarded by a credit gate inside the edge function layer.

Architecture layers

Eight distinct horizontal layers — each owns a single responsibility and communicates only through explicit API boundaries. No direct database access from the client; all data flows through the edge function or Supabase SDK.

L1
UI — Flutter / Dart Screens, widgets, animations. Material 3 tokens + custom design system from core_ui. Flutter
L2
State — Riverpod 2 Async state management, dependency injection. AsyncNotifier / StreamProvider per module. Riverpod
L3
Navigation — GoRouter 14 Deep links, nested shell routes. Each feature module registers its routes into the root shell. GoRouter
L4
Network — Dio + core_network HTTP client, retry, token-refresh interceptor. Typed NetworkException hierarchy maps HTTP codes. Dio
L5
API Gateway — Supabase Edge Fn JWT-verified Deno functions handle billing webhooks, AI proxying, and credit gating. Keys never reach the client. Deno
L6
Database — PostgreSQL / RLS All 12 user-facing tables enforce Row-Level Security. auth.uid() scopes every read/write to the requesting user. RLS
L7
Storage — Supabase Storage Audio assets (speaking recordings), images, PDF exports. Bucket policies restrict per content type. S3-compatible
L8
Auth — Supabase Auth Identity, session management, short-lived JWTs. Magic-link email + OAuth (Google, Apple). No password storage. JWT

Layer specification

Each horizontal layer owns a distinct responsibility. Cross-layer calls are explicit API boundaries — there is no direct DB access from the client.

Request flow walkthrough

Every AI-powered action follows the same security pipeline. The Flutter client is the only user-facing artifact — no credentials are stored on device.

1
User triggers action
User taps a UI element (e.g. "Evaluate my essay"). A Riverpod AsyncNotifier method is called. L1 → L2.
2
Network request dispatched
core_network Dio client sends a POST to the Supabase Edge Function URL. The JWT from Supabase Auth is attached by the AuthInterceptor. L4 → L5.
3
Edge function: identity + credit check
Deno function verifies the JWT via _shared/auth.ts, extracts user_id, then queries user_credits.balance. Insufficient balance returns 402. L5 → L6.
4
Upstream AI provider called
The edge function calls the AI provider (Gemini, Deepgram, ChatGPT-4o, or Inworld) using server-side API keys. Up to 10 retries with exponential backoff. L5 → external.
5
Credits deducted, history written
On success: UPDATE user_credits (deduct), INSERT practice_history (log session). Both writes happen in a single transaction. L5 → L6.
6
Response delivered to client
Edge function returns structured JSON. The Riverpod notifier updates state, Flutter rebuilds the UI. L5 → L4 → L2 → L1.

Security model

Security is layered — each concern is addressed at the most appropriate layer, with no single point of failure.

N
No client API keys All AI provider keys (Gemini, Deepgram, Inworld, OpenAI) and billing secrets (App Store shared secret, Google service account, Lemon Squeezy signing key) are stored exclusively in Supabase Edge Function environment secrets. The Flutter client never holds or transmits a third-party API key.
Authentication
  • Short-lived JWTs issued by Supabase Auth, automatically refreshed by AuthInterceptor
  • Magic-link + OAuth (Google, Apple) — no password storage
  • Every edge function call requires a valid JWT before processing
Database security
  • Row-Level Security enabled on every table
  • All policies use auth.uid() — users can only read/write their own rows
  • Content tables (questions, passages) are SELECT-only for authenticated users
  • Billing nonces table is service-role only (no direct client access)
Billing anti-fraud
  • Every purchase receipt is server-verified with the issuing platform
  • billing_nonces table prevents replay attacks and double-credit grants
  • Credit grants only execute after platform API confirms validity
Storage security
  • Supabase Storage bucket policies restrict access by user and content type
  • Signed URLs used for temporary audio asset access
  • User-uploaded audio (speaking practice) is scoped to the uploading user

Design decisions

D
Why Supabase? Supabase provides auth, database (PostgreSQL), storage, and serverless compute in one platform. This eliminates the need to wire together separate services, reduces vendor count, and gives strong RLS guarantees out of the box. Edge Functions (Deno) run on the same network as the database, minimising latency for credit-check + AI-call sequences.
D
Why Flutter? A single Dart codebase targets iOS, Android, and web from one source tree. The 14-module architecture means TOEFL, SAT, IELTS, and GRE variants share the same core_ui, core_network, and core_billing packages — only feature modules differ per app.
T
Modular by design Each feature module is independently testable and deployable. Riverpod providers are the only cross-module communication channel — direct class imports between modules are prohibited. This makes it safe to add or refactor a module without risking regressions in others.
